## Virtual Reality in the Web
VR is a new media.
For VR to take off, we need **content** and content requires authors and distribution.
The Web and Web technologies are free, frictionless and reach millions. They seems the [perfect choice for such a mission](../web-platform-when-why/).
WebVR is an initiative started by Mozilla for bringing VR experiences to the Web.
[Initial implementations](http://web.archive.org/web/20151016091437/http://blog.bitops.com/blog/2014/06/26/first-steps-for-vr-on-the-web) were a hack on the Fullscreen API in Firefox and Chrome experimental versions.
These experiments lead the to **WebVR 1.0**.
The [WebVR sepecficiation](https://w3c.github.io/webvr/) is edited by the [Mozilla MozVR team](https://mozvr.com/) and Google, and enables access to VR devices, including head-mounted displays, sensors and controllers.
Browsers supporting WebVR: * [Firefox Nightly](https://nightly.mozilla.org/) * [Chromium](https://webvr.info/get-chrome/) * [Samsung Internet](http://developer.samsung.com/internet#gearvr-overview)
Companies like Samsung, Microsoft and Oculus are joining the effort to help improving the standard.
So now we have a Web standard, it should be as easy as...
* Getting the VR device * Getting details for each eye * Creating the 3D context * Getting pose * Painting the scene once per eye ...
* Providing polyfills * Dealing with fullscreen * Dealing with mobile * Dealing with resize ...
...oh WAIT!
## A-Frame HTML for Virtual Reality
A-Frame is a web framework for **building VR experiences**. It drastically reduces the boilerplate to:
```html <a-scene></a-scene> ```
A-Frame uses [Custom Elements](http://jonrimmer.github.io/are-we-componentized-yet/) to extend the HTML vocabulary enabling the use of new **tags for the 3D scene management** such as `a-scene`, `a-box` or `a-camera`.

Scene manipulation is achieved by modifying the DOM.

See the Pen A-Frame Scene Manipulation by Salvador de la Puente González (@lodr) on CodePen.

A-Frame provides general primitives for building scenes: * [`a-scene`](https://aframe.io/docs/0.4.0/core/scene.html) is the root of the 3D scene. * [`a-animation`](https://aframe.io/docs/0.4.0/core/animations.html) allows animation of scene elements. * `a-light` creates lights emitters. * `a-camera` instantiates a camera in the scene.
* `a-sound` places positional audio. * `a-videosphere` for playing 360º videos. * `a-curvedimage` displays curved images for improved readability. * `a-plane`, `a-box`, `a-sphere`... as geometry primitives. Get the complete list of tags at the [primitive documentation](https://aframe.io/docs/0.4.0/primitives/).
Since A-Frame works on the top of HTML, integration with other frameworks or libraries such as [React](https://github.com/ngokevin/aframe-react) or jQuery is easy, requiring from none to just a thin adaptation layer.

## Entities, Components & Systems
The real power of A-Frame lies in its [ECS architecture](https://aframe.io/docs/0.4.0/core/).
### E is for Entity An [entitiy](https://aframe.io/docs/0.4.0/core/entity.html) is an element of the scene and a container for components. ```html <a-entity></a-entity> ```
### C is for Component A [component](https://aframe.io/docs/0.4.0/core/component.html) is data associated to an element. ``` <a-entity position="0 2 -5"></a-entity> ``` Review the [complete list of components](https://aframe.io/docs/0.4.0/components/)!
Components can store multiple data using a sort of CSS syntax. ```html <a-entity position="0 2 -5" camera="near: 0.1; far: 100"></a-entity> ```
### S is for System A [system](https://aframe.io/docs/0.4.0/core/systems.html) is the logic governing entities' components.
All A-Frame is based on the ECS pattern. Even primitives are actually entities whose attributes map to specific components.
So, the primitive: ```html <a-box color="red"></a-box> ``` is actually a shortcut for: ```html <a-entity geometry="primitive: box" material="color: red"></a-entity> ```

## 3rd Party Components
Components are the main point of extension in A-Frame. Some are built-in in A-Frame:
`camera`, `cursor`, `debug`, `embedded`, `geometry`, `ligth`, `position`, `rotation`, `scale`, `sound`, `visible`, `tracked-controls`, `obj-model`...
But developers can create their own components:
`alongpath`, `animation`, `mountain`, `outline`, `particle-system`, `physics`, `teleport-controls`, `auto-detect-controllers`, `bmfont-text`, `crawling-cursor`, `deferred-renderer`, `firebase`, `gearvr-controls`, `layout`, `speech-command`, `stats-in-vr`, `stereo`, `super-hands`...
And you can find them all in the [A-Frame Registry](https://aframe.io/aframe-registry/)!
Using a 3rd party component is as simple as including the component file. ```html <script src="https://aframe.io/releases/0.5.0/aframe.min.js"></script> <script src="https://rawgit.com/IdeaSpaceVR/aframe-particle-system-component/master/dist/aframe-particle-system-component.min.js"></script> ```

For instance, there is no particle system in the core library.
